## 1. Резюме

Рестор на свежую установку ГУ вернул поездки, у которых **поля внутри одной строки принадлежат разным физическим поездкам**. Затронуты 53 строки из 86 (локальные id 2–52, 54–55) — это все поездки, существовавшие на сервере до ~04.07 22:21 Minsk (19:21 UTC). Всё, что создано после, — чистое.

Клиентский код применения pull/restore исключён как источник (см. §3.1) → **строки испорчены в самом серверном каноне**. Порча невидима с сервера «на глаз»: количество строк не менялось, дублей нет, каждая строка выглядит валидной поездкой.

## 2. Симптоматика (что именно перемешано)

По 53 испорченным строкам:

- `started_at` по локальному id идёт **строго по возрастанию**, а `client_uuid` (время, зашитое в UUIDv7) — **строго по убыванию**. Пример: строка со `started_at` 25.05 10:48 несёт uuid, отчеканенный 04.07 15:45; строка со `started_at` 03.07 09:51 несёт uuid от 29.05 11:51.
- **23 строки с `ended_at < started_at`** (id 26–48): отрицательная длительность, физически невозможно.
- **Агрегаты задублированы зеркальными парами** внутри id 2–48 (пары в сумме дают 50): (2↔48), (3↔?), (4↔46), (5↔45), (6↔44), (7↔43), (8↔42), (9↔41), (10↔40), (11↔39), (12↔38), (13↔37), (14↔36), (15↔35), (16↔34), (17↔33), (18↔32), (19↔31), (20↔30), (21↔29), (22↔28), (23↔27), (24↔26). У каждой пары `distance_km` и `sample_count` идентичны → у одной поездки из каждой пары настоящие агрегаты **перезаписаны и потеряны** (в каноне; надежда только на бэкап).
- Подгруппа id 49–52, 54–55: поля строки (start/end/агрегаты) внутренне **когерентны** (это реальные поездки 03–04.07 целиком), но uuid — чужой (от майских поездок). Т.е. здесь замещение было **построчным целиком**, а в зоне 2–48 — **пополам** (start-половина от одной поездки, end-половина от другой).

## 3. Доказательства локуса (сервер, не клиент)

### 3.1 Клиент не может смешать поля
В дереве +144 `_applyPulledTrip` → `_tripCompanionFromJson(data)`: каждая строка (и в restore, и в 5-мин `_syncPull`) собирается из **одного** JSON-объекта. Пути, где started_at берётся из одного объекта, а ended_at из другого, в коде нет. Finalize-UPDATE тоже пишет companion из одного объекта (с `clientUuid: absent`).

### 3.2 Снапшоты того же pull — идеально чистые
`snapshots` (insert-only сущность на сервере): 1423 строки с одометром, **0 инверсий одометра по captured_at** за весь диапазон 25.05→16.07. Тот же транспорт, тот же рестор, та же сессия → транспорт/применение исправны. Порча специфична для trips — единственной **UPSERT**-сущности.

### 3.3 Обе «половины» испорченной строки — реальные данные разных поездок
Сверка с чистыми снапшотами (полные данные в CSV, колонки `snap_odo_at_*`):
- `start_odometer` строки совпадает со снапшотом одометра на момент `started_at` — **39/39 совпадений** там, где снапшот в окне ±15 мин существует, 0 расхождений.
- `end_odometer` строки совпадает со снапшотом на момент `ended_at` — **26/26 совпадений**, 0 расхождений.

Т.е. (started_at + start_odometer) — настоящая пара поездки A, (ended_at + end_odometer) — настоящая пара поездки B, склеенные в одну строку. Контрольный пример пары (24, 26): строка 24 start_odo 3343.8 (04.06 17:18), dist 14.8; строка 26 end_odo **3358.6** (04.06 18:15) = 3343.8 + 14.8 — конец той самой поездки A лежит на зеркальной строке.

### 3.4 Граница совпадает со штормом 04.07 с точностью до минуты
Последняя испорченная строка несёт uuid от 04.07 18:24 (id 55). Первая чистая поездка — **id 56, старт 04.07 22:22:03 Minsk**. Задокументированный шторм re-push всего локального архива: **04.07 19:21 UTC = 22:21 Minsk**. Поездка, стартовавшая через минуту после шторма, уже чиста.

## 4. Гипотеза механизма

Та самая мина из спеки cloud v2 («НЕ wipe-ать локальную Drift-БД: trips = UPSERT по (device_id, client_trip_id); wipe рестартует autoincrement → новые поездки молча перезаписывают старые»), сработавшая в два слоя:

1. **04.07 19:21 UTC** — re-push всего локального архива со сброшенными курсорами, гейт push v2 OFF → `client_uuid=NULL` в пуше → сервер матчил по legacy-ключу `(device_id, client_trip_id)`. Локальные id после рестора были перенумерованы → пуш поездки B прилетел в серверную строку поездки A → поля перезаписаны, uuid строки остался родной. Это объясняет подгруппу 49–55 (целиком чужие поля при родном uuid).
2. **05.07 15:21 UTC** — гонка restore↔sync, ещё 254 legacy-строки с другим сдвигом id → второй слой перезаписи поверх первого. Композиция двух перестановок с разными наборами колонок и объясняет «половинчатую» зону 2–48.

Снапшоты от тех же штормов дали **видимые дубли** (insert-only) — их ты вычистил 05.07. Trips-UPSERT дублей не создал, порча осталась невидимой. Клиентские источники закрыты давно (+123/+126, push v2 uuid-дедуп) — **новых повреждений после 05.07 нет**, зона стабильна.

Открытый вопрос механики (для точной форензики): в зоне 2–48 started-половина и ended-половина разъехались **внутри строки** — проверь, что именно делает trips `ON CONFLICT DO UPDATE SET` (все ли колонки, или started_at/часть колонок исключены/COALESCE) — от этого зависит, какой из двух штормов какую половину положил.

## 5. Странности/исключения

- Две поездки канона стартуют в одну секунду 25.05 10:48:51 (локальные id 2 и 3) — известные легаси-призраки, uuid разные, дедупу не мешают, просто имей в виду при матчинге.

## 6. План починки (предложение)

### 6.1 Основной путь — offsite-бэкап
Взять Scaleway-бэкап с дампом **до 04.07 19:21 UTC** → у всех затронутых поездок там должны быть целые строки. Матчить по `client_uuid` (они в каноне не менялись — только поля вокруг них): `UPDATE canon SET <все поля> FROM backup WHERE canon.client_uuid = backup.client_uuid` для затронутого множества. Поездки 03–04.07, родившиеся между бэкапом и штормом (если бэкап старее), — восстановить нечем из бэкапа, но их поля целиком лежат на строках 49–55 текущего канона (когерентная подгруппа) — их достаточно перевесить на правильные uuid.

### 6.2 Fallback — реконструкция без бэкапа (если бэкап не покрывает)
Данных для восстановления хватает в самом каноне + чистых снапшотах:
- Множества настоящих `started_at` и настоящих `ended_at` целы (просто раскиданы по чужим строкам).
- Одометр монотонен → настоящие пары восстанавливаются интерливингом: для каждого (started_at, start_odo) настоящий конец = минимальный (ended_at, end_odo) с end_odo ≥ start_odo, не залезающий за следующий старт. Контроль: end_odo − start_odo ≈ distance_km (подтверждено на паре 24/26).
- Оракул валидации любого ремонта: снапшоты (captured_at + одометр) — восстановленная поездка обязана совпадать со снапшотами на своих краях.
- Агрегаты зеркальных пар: у одной строки пары они настоящие, у второй — потеряны; без бэкапа вторые не вернуть (пометить или пересчитать грубо из снапшотов ΔSOC/Δodo).

### 6.3 Самолечение клиентов — автоматом, без переустановок
После починки канона данные строки должны **бампнуть seq** (твой UPSERT «no-op не бампает seq» — реальный ремонт no-op'ом не будет). Клиенты подтянут их штатным 5-мин pull: finalize-UPDATE по uuid-хиту (`ended_at≠null` → UPDATE всех полей кроме identity) перепишет локальные строки. Ни ГУ, ни телефон трогать не нужно. Клиентского патча не требуется.

### 6.4 Порядок
pg_dump текущего канона (до любых правок) → сверка с бэкапом (dry-run diff по затронутым uuid, counts) → боевой UPDATE одной транзакцией → smoke: у случайных 5 uuid started/ended/odo сходятся со снапшотами → убедиться, что seq бампнулся → сказать Alex, чтобы дождался pull на обоих устройствах и глянул историю за май–июнь.
