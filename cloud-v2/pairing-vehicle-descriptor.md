# Контракт-дополнение: `vehicle` при pair/start (self-service онбординг)

**Дата:** 2026-07-12 · **Автор:** Друг 2 (bridge, carbridge) · **Статус:** ✅ server deployed (prod head 0011)
**Связано:** `account-approval-server-ready.md`, SPEC Друг 1. **Это новый пункт клиентского контракта — нужен в том же релизе, что и +133 (или сразу +134).**

## Зачем

Новый пользователь из стора **не имеет машины** в облаке. А `pair/claim` требует vehicle, принадлежащий аккаунту. Раньше это работало только потому, что у Alex машина засеяна в БД; у нового юзера её неоткуда взять. Поэтому добавлен self-service provisioning: **клиент при pairing сообщает, что за машина, и сервер её заводит.**

Без этого блока новый аккаунт после апрува получит `400 no_vehicle` на claim и не сможет синхронизироваться. **Существующие устройства Alex не затронуты** (у них уже есть машина — блок игнорируется).

## Что слать

На **`POST /v2/pair/start`** добавь опциональный блок `vehicle`:

```json
POST /v2/pair/start
{
  "kind": "headunit",
  "client_kind": "flutter-bz5-companion",
  "display_name": "BZ5 head unit",
  "hw_fingerprint": "<Android_ID, как раньше>",
  "vehicle": { "make": "BYD", "model": "Seal", "generation": "2024", "name": "Моя Seal" }
}
```

- `make`, `model` — **обязательны внутри блока**, хранятся **как есть**.
- `generation`, `name` — опциональны (`name` по умолчанию = `"{make} {model}"`).

## ⚠️ Мульти-мейк — не хардкодь Toyota

DiLink 5.0 стоит **не только в Toyota bZ, но и в BYD и др.** Поэтому `make`/`model` — это **реальная марка/модель с устройства или от пользователя**, а не константа. Сервер хранит их дословно и никогда не подставляет бренд. Если приложение умеет определять марку с ГУ — бери оттуда; иначе спроси пользователя при первом подключении к облаку.

## Когда слать

- **Новый аккаунт, свежая привязка (сценарий b):** слать `vehicle` обязательно — иначе `claim` → `400 no_vehicle`.
- **Аккаунт уже с машиной** (реинсталл, второе устройство, Alex): блок можно не слать — он игнорируется, привязка идёт к существующей машине.
- Правило простое: **всегда клади `vehicle` в pair/start, если он у тебя есть** — сервер сам решит, создавать машину или использовать имеющуюся. Лишним не будет.

## Поведение сервера (уже задеплоено)

| Ситуация | Результат claim |
|---|---|
| Аккаунт без машины + есть `vehicle` | создаётся машина (owner = аккаунт), устройство привязывается ✅ |
| Аккаунт без машины, `vehicle` не прислан | `400 no_vehicle` |
| Аккаунт с 1 машиной | привязка к ней, `vehicle` игнор (как раньше) |
| Аккаунт с >1 машиной | нужен явный `vehicle_id` (мульти-авто вне scope) |

`pair/status` без изменений (отдаёт `device_id` + `client_token` + `attach_mode`). Один аккаунт = одна машина (мульти-авто пока вне scope).

## Итог / порядок

- Сервер готов (миграция 0011 в бою).
- От тебя: слать `vehicle{make,model,…}` на pair/start. Это **обязательно для онбординга новых юзеров** → должно войти в релиз с account-кодами (+133) либо в ближайший +134.
- После этого: устройства Alex обновляются → Alex флипает `APPROVAL_GATE=on` → публикация стора.

Полная схема запроса — в CLIENT_API.md §1.2 (`https://carbridge.neardo.work/client-api.md`).
