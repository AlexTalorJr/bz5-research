# Сервер готов — можно выпускать клиент +133 (account approval)

**Дата:** 2026-07-12 · **Автор:** Друг 2 (bridge, carbridge) · **Статус:** ✅ server-side COMPLETE, deployed
**Продолжение:** `account-approval-server-response.md` (контракт), SPEC Друг 1 «Ручной апрув аккаунтов».

Вся серверная часть ручного апрува аккаунтов собрана, задеплоена и покрыта тестами (145 зелёных, prod migration head 0010). **`APPROVAL_GATE` сейчас `off` → поведение продакшена байт-в-байт как раньше**, коды `account_*` в проде ещё не появляются. Можно спокойно выпускать **+133** — он безопасен при выключенном гейте и активируется, когда владелец флипнет гейт (строго после того, как +133 будет на устройствах).

## Что задеплоено

| Часть | Итог |
|---|---|
| `users.status` (`pending`/`approved`/`rejected`/`blocked`) + бэкфилл | существующие аккаунты → `approved`, ничего не замечают |
| allowlist → **auto-approve list** | owner/allowlisted при первом входе → сразу `approved`; остальные (при gate on) → `pending` |
| **data-plane гейт** | `/v1/data/ingest/*` + весь `/v2/sync/*` → 403 `account_*` для не-approved; account-plane открыт |
| admin API `/v1/admin/accounts/*` | approve/reject/block/unblock (owner-side) |
| **Telegram-бот** | живой: владелец апрувит кнопками ✅/❌ прямо в чате |
| rate-limit регистраций + автопурж pending | см. ниже |

## Контракт для клиента — подтверждён, без изменений

То, что мы зафиксировали в `SPEC_plus133 §0.1`, в силе и совпадает с тем, что реально отдаёт прод:

- **Формат 403 по проводу:** `{"detail":{"error":{"code":"account_pending|account_rejected|account_blocked","message":…}}}` — та же оболочка, что `samples_disabled`. Парсер `detail.error.code` (`cloud_sync_service.dart:2455`) не меняется; три кода — до ветки `samples_disabled`. Все три → permanent-no-retry.
- **Гейтятся:** `/v1/data/ingest/*` (incl. heartbeat), `/v2/sync/*`. **Открыты для pending:** `/v2/auth/*`, `/v2/devices*`, `/v2/pair/*` — устройство пары́тся и юзер логинится, ожидая апрув.
- **Апрув мгновенный:** статус читается из БД на каждый запрос — после approve следующий тик ingest/pull проходит 200 без действий на клиенте.
- `/v1/setup/*` **не** гейтится (owner-only SETUP-токен, не end-user; решение владельца).

## Одно небольшое дополнение к контракту (учти в +133)

**Rate-limit регистраций: 3 новых аккаунта с одного IP в сутки** (только при gate on). Превышение → `429` с `detail.error.code == "rate_limited"` на `POST /v2/auth/otp/verify`. Нюанс: **OTP при этом уже потрачен**, так что для повторной попытки нужен свежий код. Вход в *существующие* аккаунты не задет — лимит только на создание новых. Обработка — как любой другой `429`: показать пользователю «слишком много попыток, попробуйте позже», не спамить ретраями.

(Автопурж брошенных pending старше 30 дней — чисто серверный; для клиента это значит лишь: удалённый pending → следующий OTP просто заводит новый pending.)

## Порядок выката (напоминание §7)

1. ✅ Сервер задеплоен, `APPROVAL_GATE=off`.
2. **→ Ты выпускаешь +133**, владелец обновляет ГУ + телефон.
3. Только потом владелец флипает `APPROVAL_GATE=on` (+ открывает стор).

Пока гейт off, старый клиент и +133 ведут себя идентично (коды `account_*` не появляются). Так что релиз +133 ничем не рискует — он просто «готов заранее».

## Итог

Со стороны сервера **всё готово, блокеров нет**. От тебя — выпустить +133 по уже согласованному контракту (+ обработать `429 rate_limited` на otp/verify). Как выйдет и встанет на устройства — владелец включит гейт, и я проверю предусловия перед флипом.
