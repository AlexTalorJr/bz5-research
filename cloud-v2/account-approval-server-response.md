## 1. 🔑 Точная форма 403 — критично, свери свой парсер +133

Спека рисует `403 {"error":"account_pending"}` (плоско). **Реальный формат по проводу другой** — все ошибки, кроме валидации схемы, идут вложенно под `detail`:

```json
403 {"detail": {"error": {"code": "account_pending", "message": "…"}}}
403 {"detail": {"error": {"code": "account_rejected", "message": "…"}}}
403 {"detail": {"error": {"code": "account_blocked",  "message": "…"}}}
```

Парсить надо **`body.detail.error.code`** — ровно как ты уже парсишь `samples_disabled` (это твоя референс-точка, и она приходит в **точно такой же** оболочке; проверено в бою на carbridge). Ветку распознавания трёх `account_*` ставь **до** `samples_disabled`, как и планировал. Все три → permanent-no-retry (403), без Retry-After.

Если +133 уже написан на плоский `{"error":...}` — поправить на `detail.error.code`, иначе не распознает. Это **единственное**, что жёстко связывает наши два патча — фиксируем как контракт.

## 2. Множества эндпоинтов — твой список верен, точные пути

**Гейтятся (403 `account_*` при не-approved):**
- `/v1/data/ingest/*` (все ingest, включая heartbeat)
- `/v2/sync/*` (`uuid-mapping`, `pull`)
- `/v1/setup/*` (register-device, vehicles)

**НЕ гейтятся (работают для pending как раньше):**
- `/v2/auth/*` (otp/request·verify·refresh·logout)
- `/v2/devices` + `/v2/devices/{id}/revoke`
- `/v2/pair/*` (start·claim·status)

Устройство может **полностью спариться заранее**; синк вернёт `account_pending`, пока Alex не апрувнет.

## 3. Апрув действует мгновенно, без действий на клиенте

Гейт читает статус аккаунта **из БД на каждом data-plane запросе** (не из JWT-claims). После approve **следующий же плановый тик ingest/pull проходит 200** — клиенту ничего перезапрашивать/перелогиниваться не надо. Симметрично: block ранее approved → следующий запрос сразу 403 (не ждём протухания JWT ≤15 мин).

## 4. heartbeat под гейтом

Heartbeat гейтится → у pending-устройства вернёт `account_pending`. Трактуй как остальные data-plane 403 (permanent, ждём следующего тика, без шторма). `last_seen` у pending на сервере не обновляется — косметика admin-стороны.

## 5. Решения Alex по §9

- **Q1 — 3 новых pending с одного IP в сутки** (не 5). Свыше — регистрация мягко отклоняется. Реализация серверная (DB-backed по `auth_events.ip`), клиента не касается.
- **Q2 — автопурж pending 30 дней.** `rejected`/`blocked` НЕ пуржатся (иначе отказ обходится перерегистрацией). Клиенту: удалённый pending = следующий OTP просто заводит новый pending.
- **Q3 — long-poll** (`getUpdates`), без webhook/nginx-правок. Инфраструктура серверная.

## 6. Хендшейк по feature-флагу (порядок §7)

Чтобы не застрелить старых клиентов:
1. Я деплою всё с **`APPROVAL_GATE=off`** — поведение байт-в-байт как сейчас (существующие аккаунты бэкфиллятся в `approved`).
2. **Ты выпускаешь +133**, Alex обновляет ГУ+телефон.
3. Только после этого Alex флипает **`APPROVAL_GATE=on`**.

Старый клиент (<+133) под включённым гейтом получит generic `CloudSyncStatus.error` с телом 403 в `lastError` — некрасиво, но безопасно (данные локальны, 403=permanent → без ретрай-шторма). Твоя оценка §7 верна. Флип `on` — строго после того, как Alto-устройства на +133.

## 7. Фаза 2 (инвайт-коды) — контракт принят

`POST /v2/auth/otp/verify` + опциональное `invite_code`; валидный → мгновенный `approved`; невалидный/просроченный → **игнор, не ошибка** (аккаунт остаётся pending, анти-энумерация кодов). Реализую после обкатки ручного апрува. От клиента в этой фазе — только поле `invite_code` в verify, когда дойдём.

Единственное изменение на твоей стороне — **парсить `detail.error.code` и добавить три `account_*` кода перед `samples_disabled`**. Всё остальное серверное. Как подтвердишь формат из §1 — стартую с миграции 0010.

## Приложение: серверные решения (не требуют действий клиента)

- **Таблица — `users`** (не `accounts`): статус-колонки вешаются на неё.
- **Allowlist → auto-approve list** (решение Alex 2026-07-11): allowlisted/owner при первом verify → сразу `approved` (`status_changed_by='auto'`), остальные → `pending`. Открывает регистрацию всем, сохраняет мгновенный доступ известным адресам, стыкуется с инвайтами Фазы 2. Заменяет прежний смысл D10 (allowlist-как-гейт).
- **Admin API** — под существующим `/v1/admin/*` + существующий `ADMIN_TOKEN_HASH` (новый токен не вводится).
- **Формат ошибок гейта** переиспользует существующий `_forbidden`/`_err` (та же оболочка, что `samples_disabled`).
- **Бот** — best-effort, вне request-path (падение TG не ломает OTP-verify).
- **Rate-limit** — DB-backed по `auth_events.ip` (наши in-memory счётчики сбрасываются на деплое → для суточного окна не годятся).
- **Порядок ввода** §7 принят целиком; миграция **0010**.
